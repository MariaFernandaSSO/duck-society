Condicionais: if, else e switch — seu código tomando decisões

Conditionals: if, else and switch — your code making decisions

Até agora nossos exemplos eram lineares: faz uma coisa, depois outra, depois acaba. Mas a vida real não é linear. "Se chover, levo guarda-chuva. Se não, óculos de sol." No código, isso se chama condicional.

if / else if / else

A estrutura mais básica de decisão. O computador avalia uma expressão booleana e executa um bloco ou outro.

Java:

int temperatura = 30;

if (temperatura > 35) {
    System.out.println("Muito quente, fique em casa");
} else if (temperatura > 25) {
    System.out.println("Clima agradável");
} else if (temperatura > 15) {
    System.out.println("Fresco, leve um casaco");
} else {
    System.out.println("Frio!");
}

Go:

temperatura := 30

if temperatura > 35 {
    fmt.Println("Muito quente, fique em casa")
} else if temperatura > 25 {
    fmt.Println("Clima agradável")
} else if temperatura > 15 {
    fmt.Println("Fresco, leve um casaco")
} else {
    fmt.Println("Frio!")
}

Armadilhas comuns

  • Usar = em vez de == em Java: if (idade = 18) não compila em Java (diferente de C, que aceita). Mas em Go, if idade = 18 também não compila. Seguro :D
  • Esquecer o else quando precisa: às vezes você quer um comportamento padrão. Se não tem else, e nenhum if entra, simplesmente nada acontece — e pode ser um bug silencioso.
  • Condição muito complexa: if (a && b || !c && d) é impossível de ler. Extraia pra variáveis com nomes.

Exemplo real: validação de CPF (simplificado)

String cpf = "12345678909";

// Primeiro: validar tamanho
if (cpf == null || cpf.length() != 11) {
    System.out.println("CPF inválido: deve ter 11 dígitos");
} else if (cpf.matches("(\\d)\\1{10}")) {
    System.out.println("CPF inválido: todos dígitos iguais");
} else {
    System.out.println("CPF válido (ignorando dígitos verificadores por enquanto)");
}

Exemplo real: status de pedido

String status = "ENVIADO";

if (status.equals("PENDENTE")) {
    System.out.println("Aguardando pagamento");
} else if (status.equals("PAGO")) {
    System.out.println("Preparando para envio");
} else if (status.equals("ENVIADO")) {
    System.out.println("Pedido a caminho");
} else if (status.equals("ENTREGUE")) {
    System.out.println("Pedido finalizado");
} else {
    System.out.println("Status desconhecido");
}

Repare como equals() em Java é usado pra comparar strings, não ==. Isso porque == compara referência, não conteúdo.

switch — quando if vira bagunça

Quando você tem muitas condições em cima da mesma variável, if vira uma escada feia. switch foi feito pra isso.

Java (switch tradicional):

int diaSemana = 3;
String nomeDia;

switch (diaSemana) {
    case 1:
        nomeDia = "Domingo";
        break;
    case 2:
        nomeDia = "Segunda";
        break;
    case 3:
        nomeDia = "Terça";
        break;
    case 4:
        nomeDia = "Quarta";
        break;
    case 5:
        nomeDia = "Quinta";
        break;
    case 6:
        nomeDia = "Sexta";
        break;
    case 7:
        nomeDia = "Sábado";
        break;
    default:
        nomeDia = "Inválido";
}

System.out.println(nomeDia);

Java (switch moderno — Java 14+):

int diaSemana = 3;
String nomeDia = switch (diaSemana) {
    case 1 -> "Domingo";
    case 2 -> "Segunda";
    case 3 -> "Terça";
    case 4 -> "Quarta";
    case 5 -> "Quinta";
    case 6 -> "Sexta";
    case 7 -> "Sábado";
    default -> "Inválido";
};

System.out.println(nomeDia);

Go (switch é mais simples e não precisa de break):

diaSemana := 3
var nomeDia string

switch diaSemana {
case 1:
    nomeDia = "Domingo"
case 2:
    nomeDia = "Segunda"
case 3:
    nomeDia = "Terça"
case 4:
    nomeDia = "Quarta"
case 5:
    nomeDia = "Quinta"
case 6:
    nomeDia = "Sexta"
case 7:
    nomeDia = "Sábado"
default:
    nomeDia = "Inválido"
}

fmt.Println(nomeDia)

Armadilha: fallthrough

Em Java (tradicional), se você esquece o break, o switch "cai" pro próximo caso — isso se chama fallthrough. Muitas vezes é um bug. Por isso Java moderno introduziu a sintaxe com -> (que não tem fallthrough), e Go já nasceu sem fallthrough automático.

Comparativo if vs switch

  • if: melhor para condições complexas (intervalos, combinações de variáveis)
  • switch: melhor para uma variável com muitos valores conhecidos
  • Nenhum é "superior"; cada um tem seu lugar

Resumo

  • if / else if / else é a base de toda decisão no código
  • switch organiza múltiplos casos da mesma variável
  • Cuidado com fallthrough em Java tradicional (não esqueça o break)
  • Extraia condições complexas pra variáveis com nomes legíveis
